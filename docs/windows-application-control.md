# Windows Application Control error 4551

## Symptom

Cargo fails with an error like:

```text
could not execute process ...\build-script-build
An Application Control policy has blocked this file. (os error 4551)
```

This can happen for dependencies such as `serde_json`, `zmij`, `parking_lot_core`, or any crate that has a `build.rs`.

## Cause

This is not a Rust source error. Windows Defender Application Control, Smart App Control, or an enterprise WDAC policy is blocking unsigned executables generated by Cargo during compilation.

Rust builds commonly compile and execute small helper binaries named `build-script-build`. Tauri cannot build while those generated binaries are blocked.

## First workaround

The dev runner sets `CARGO_TARGET_DIR` outside the repository:

```powershell
.\dev.ps1
```

By default it prefers:

```text
C:\tmp\meetings-recorder-cargo-target
```

You can set it manually:

```powershell
$env:CARGO_TARGET_DIR="C:\tmp\meetings-recorder-cargo-target"
.\dev.ps1
```

## If it still fails

If `C:\tmp` also fails, the policy is blocking Cargo-generated executables regardless of target location. The practical fixes are:

- Turn off Smart App Control for this development machine.
- Ask IT/security to allow Rust/Cargo build artifacts for this project.
- Build on a machine without WDAC restrictions.
- Use WSL for Rust-only validation, noting that the Windows Tauri desktop app still needs a Windows build for the final executable.

## What does not fix it

- Changing `serde_json`, `zmij`, or other dependency versions usually does not solve the class of problem.
- `Unblock-File` usually does not help because these files are generated locally and blocked by policy, not by Mark-of-the-Web.
- Cleaning `target/` only causes Cargo to regenerate another blocked executable.
